Intro to Git Bash:

- What is Git Bash?
Git Bash is a *version control system (VCS)*, meaning it's a software designed to record changes within one or more files over time. It also allows us to undo - or even cancel - the made or pending changes in one or more files.

You might be wondering, what does this mean? Why does this matter to me?

Well, using a VCS (in this case, Git Bash) can help you track your projects.

Say you were working on a webpage, and you accidentally deleted or erased the contents of the main page. You would then lose the main file in your project, as well as hours of work.

But, if you used Git Bash to track these files and changes made to said files, you could simply revert the changes (more on that later).


- What is the command line?
The command line is an interface for typing commands directly to a computer's operating system.

Every Windows PC comes with a Command Prompt (a type of command line), which allows you to install programs, create new users, and run scripts.


- What's the difference between Git Bash and the command line?
The difference between Git Bash and the command line is simple - Git Bash is a TYPE OF command line, not THE command line. 

Kind of like the difference between vanilla ice cream and ice cream - there isn't really a difference, vanilla is simply a flavor of ice cream. Mmmm, ice cream...


- What's the difference between the Command Prompt and Git Bash?
The difference between the Command Prompt and Git Bash is that one (the Command Prompt) is built into every Windows PC (as stated above). Bash is naturally only available for Unix-like operating systems like Linux (another operating system, or OS, like Windows). Downloading Git Bash onto your computer allows you the functionalities of Bash without changing your OS.


- Then why should I use Git Bash, not the built-in Command Prompt?
Good question. Git Bash has more powerful features, such as arithmetic, array variables, loops, and branches.

- Changing the File System
	- How do you create a new directory?
	You can use mkdir to create a new directory.

	- How do you destory a directory?
	You can use rmdir, use rm -rf to destroy it with force (no questions asked).

	- How do you create a new file?
	You can use touch to create a new file.

	- How do you destroy a file?
	You can use rm to destroy a file, rm -f to destroy it with force (no questions asked).

	- How do you rename a directory or file?
	You can use mv to rename (and move) directories and/or files. Basically...
		- mv [directory] [existing directory] moves [directory] into [existing directory]
		- mv [directory] ["new" directory] renames [directory] to ["new" directory]
		- mv [file] [directory] moves [file] into [directory]
		- mv [file] [existing file] moves [file] contents into [existing file] <- needs more testing, with actually edited files
		- mv [file] ["new" file] renames [file] to ["new" file]

Intro to Github

- What is Github
So far, you've been messing around on your computer, making changes to your file system - and only you can see those changes. You've been making changes locally.

How can you allow others to see your changes, you may ask? By pushing your changes to Github, or some other code hosting platform. That way, others can see (and even contribute to) your projects.

- Github vs. Git Bash
The difference between Github and Git Bash is that one is a version control system only (Git Bash), and one is a code hosting platform (Github). Another difference is that Git Bash will save your changes locally (only for you to see), whereas Github will save your changes remotely (for everyone to see).

*A NOTE*: Although I state that everyone can see your changes when you push them remotely, technically you can have a private repository that you still can only see (but it's saved remotely).

	- Setting Up Your Git Profile
		- How do you set your username for all repositories in Git?
			- Open Git Bash. Set a Git username: git config --global user.name "Mona Lisa". Confirm that you have set the Git username correctly: $ git config --global user.name.
		
		- How do you set your username for ONE repository in Git?
			- Open Git Bash. Change the current working directory to the local repository where you want to configure the name that is associated with your Git commits. Set a Git username: $ git config user.name "Mona Lisa". Confirm that you have set the Git username correctly: $ git config user.name.

		- How do you set your email for all repositories in Git?
			- Open Git Bash. Set an email address in Git. You can use your GitHub-provided noreply email address or any email address. $ git config --global user.email "email@example.com". Confirm that you have set the email address correctly in Git: $ git config --global user.email. Add the email address to your account on GitHub, so that your commits are attributed to you and appear in your contributions graph.

		- How do you set your email for ONE repository in Git?
			- Open Git Bash. Change the current working directory to the local repository where you want to configure the email address that you associate with your Git commits. Set an email address in Git. You can use your GitHub-provided noreply email address or any email address. $ git config user.email "email@example.com". 		- Best Practices
			- How to write a good README?
			Use this basic template for personal projects:

			Project Title: the name of your project

			Description: This is an extremely important component of the README. You should describe the main purpose of your project. Answer questions like "why did you build this project?" and "what problem(s) does it solve?". It also helps to include your motivations for the project and what you learned from it.

			Features: If your project has multiple features, list them here. Don't be afraid to brag if your project has unique features that make it stand out. You can even add screenshots and gifs to show off the features.

			How to use: Here, you should write step-by-step instructions on how to install and use your project. Any software or package requirements should also be listed here.

			Technologies: List all the technologies and/or frameworks you used and what purpose they serve in your project.

			Collaborators: If others have contributed to your project in any way, it is important to give them credit for their work. Write your team members' or collaborators' names here along with a link to their GitHub profile.
